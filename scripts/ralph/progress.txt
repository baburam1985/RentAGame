## Codebase Patterns
- Components live in `web/src/components/` with co-located test files (`ComponentName.test.tsx`)
- Data types and static data in `web/src/data/games.ts`
- Pages in `web/src/app/` (Next.js App Router, "use client" for interactive pages)
- Test files use Vitest + Testing Library with `vi.fn()` for mocks and `userEvent` for interactions
- Components use controlled props pattern (value + onChange callback lifted to parent)
- Tailwind CSS only for styling; no hardcoded pixel values
- Filter components are stateless — parent (page.tsx) owns all filter state via useState
- Docker test image uses `node:20-bookworm-slim` (Debian), NOT Alpine — Alpine lacks `xvfb`/`udev` causing build failure
- E2E playwright.config.ts: skip `webServer` when `BASE_URL` env var is set (Docker CI uses http://app:3000)
- Playwright system Chromium: set `executablePath: process.env.CHROMIUM_PATH` when that env var is present
- Sort tests: pass a `games` prop with minimal mock data to GameGrid for deterministic ordering tests; use `getAllByRole("heading", { level: 3 })` to get card names in DOM order
- GameGrid accepts optional `games` override prop — lets tests inject controlled data without mocking modules

## 2026-04-12 - US-001 - qa-failed fix (Docker CI environment)
- Root cause of CI failures (Unit Tests + E2E Tests both failed, Docker Build passed):
  - Dockerfile.test used `node:20-alpine` which does NOT have `xvfb` or `udev` Alpine packages.
    The `apk add xvfb udev` step failed with "package not found", causing the Docker image build
    to error out. Since both unit-tests and e2e-tests use Dockerfile.test, both CI jobs failed.
  - playwright.config.ts had an unconditional `webServer` that tried to start `npm run dev` on
    localhost:3000, even in Docker CI where the app runs at http://app:3000 (separate service).
- What was fixed:
  - Dockerfile.test: switched base image to `node:20-bookworm-slim` (Debian). Replaced `apk add`
    with `apt-get install chromium fonts-freefont-ttf ca-certificates`. Debian has official
    Playwright support. Corrected CHROMIUM_PATH to `/usr/bin/chromium` (Debian binary location).
  - playwright.config.ts: added `useExternalServer` flag based on `BASE_URL` env var. When
    `BASE_URL` is set, `webServer` block is omitted so Playwright doesn't try to start a local
    dev server. Also wired `CHROMIUM_PATH` env var to `executablePath` in the chromium project.
- Files changed:
  - web/Dockerfile.test
  - web/playwright.config.ts
- Gotcha: Alpine Linux (`node:*-alpine`) does NOT have `xvfb` — use `xorg-server-xvfb` or
  switch to Debian. Playwright officially doesn't support Alpine for E2E. Always use
  `node:20-bookworm-slim` or `node:20` for test images that need a browser.
- Gotcha: `CHROMIUM_PATH` is not a Playwright built-in env var — must be read and set as
  `executablePath` in the playwright.config.ts project definition.

## [2026-04-12] - US-002 - Price range slider filter
- What was implemented:
  - PriceRangeSlider component: dual-handle slider using two overlaid native `<input type="range">` elements (no external library). Includes onKeyDown handlers for ArrowRight/ArrowLeft/ArrowUp/ArrowDown to ensure Testing Library userEvent compatibility. Price range label rendered as a single template-literal text node (`$25 – $50 / day`) — do NOT use nested `<span>` elements or JSX text interpolation that splits the string, as Testing Library's getByText regex matcher requires the full string in one text node.
  - GameGrid updated: added optional `minPrice` and `maxPrice` props; all three filters (category, search, price) are AND-combined in a single `.filter()` pass.
  - page.tsx updated: added PriceRangeSlider import, PRICE_MIN/PRICE_MAX constants derived from games array, minPrice/maxPrice state initialized to full range, all passed down to GameGrid. SearchBar logic also moved inline into page.tsx (no separate SearchBar component import needed).
- Files changed:
  - web/src/components/PriceRangeSlider.tsx (new)
  - web/src/components/PriceRangeSlider.test.tsx (new, 7 tests — RED committed first)
  - web/src/components/GameGrid.tsx (modified — added minPrice/maxPrice props)
  - web/src/components/GameGrid.test.tsx (new, 13 tests including price filter tests — RED committed first)
  - web/src/app/page.tsx (modified — wired PriceRangeSlider + search state)
- Learnings/patterns discovered:
  - Testing Library `getByText` with a regex requires the matched text to live in a SINGLE DOM text node. If the display string is split across elements (e.g., `$30 <span>/ day</span>`), the regex will not match — use a template literal in a single `<span>` instead.
  - `userEvent.type(input, "{arrowright}")` fires `keydown`/`keypress`/`keyup` events but does NOT fire the native `change` event on a range input in jsdom. Must add an `onKeyDown` handler that manually calls the change callback.
  - Range input `onChange` does fire correctly when using `fireEvent.change(input, {target: {value: '30'}})`, but arrow keys require the explicit keyboard handler approach.
  - Price filter state (minPrice/maxPrice) defaults to the actual min/max of the games catalog — derive with `Math.min/max(...games.map(g => g.pricePerDay))` at module scope to avoid recalculation on every render.

## 2026-04-12 - US-001 - qa-failed fix (Docker glibc/musl binary mismatch)
- Root cause of PERSISTENT CI failures (Unit Tests failed even with Debian image):
  - The `package-lock.json` was generated on an Alpine/musl machine. npm ci is deterministic
    and installs the EXACT packages from the lock file, including platform-specific native binaries.
  - On Alpine (musl): `@rolldown/binding-linux-x64-musl` was installed (Rolldown is Vite 8's
    Rust-based bundler, required for Vitest to start).
  - On Debian glibc (Docker CI): npm ci installed the SAME musl Rolldown binary from the lock
    file. The musl binary cannot load correctly on glibc — Vitest crashes at startup before any
    test runs.
  - The 'Docker Build' CI job uses a multi-stage Dockerfile with `npm ci --omit=dev`, which
    avoids devDeps like Rolldown, so it passed. Only the test images (which install devDeps) failed.
- What was fixed:
  - Created separate Dockerfile.unit-tests (bookworm-slim, no browser) and
    Dockerfile.e2e-tests (bookworm-slim + Chromium apt package).
  - Both use `npm install --no-audit --no-fund` instead of `npm ci`. npm install resolves
    native binaries for the CURRENT platform, picking `@rolldown/binding-linux-x64-gnu` on
    Debian. This is the correct binary and Rolldown loads successfully.
  - Updated docker-compose.yml to use the new Dockerfiles per service.
- Files changed:
  - web/Dockerfile.unit-tests (new)
  - web/Dockerfile.e2e-tests (new)
  - docker-compose.yml (unit-tests and e2e-tests now use separate Dockerfiles)
- ## KEY PATTERN for all future agents:
  - NEVER commit a package-lock.json generated on Alpine/musl when Docker tests run on Debian/glibc.
  - If using npm ci in Dockerfiles: the lock file MUST be generated on the SAME platform as the container.
  - Prefer `npm install` in test Dockerfiles when cross-platform (musl dev machine → glibc CI).
  - Rolldown (Vite 8+) has mandatory native binaries — platform mismatch = silent Vitest crash.
